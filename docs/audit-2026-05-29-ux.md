## Critical — Accessibility (Priority 1)

### U1. Report button is 16×16px — violates 44×44pt minimum

**File:** `src/components/community/report-button.tsx:57`

```tsx
<button className="p-1 rounded-full text-text-secondary ...">
```

`p-1` = 4px padding. Icon = 16px. Total = 24×24px. On mobile, this is a frustration target. Users will mis-tap and accidentally hit the comment above instead.

**Fix:**
```tsx
<button className="p-2.5 -m-1.5 rounded-full text-text-secondary ...">
```

This gives 44px touch area while keeping the visual size the same (negative margin compensates).

---

### U2. Comment reply/delete buttons are text-only with no touch padding

**File:** `src/components/community/comment-list.tsx:89-104`

```tsx
<button className="text-xs text-text-secondary hover:text-accent transition-colors">
  Balas
</button>
<button className="text-xs text-text-secondary hover:text-bearish ...">
  Hapus
</button>
```

No padding at all. Touch target is the exact width of the word "Balas" (~30px wide × ~16px tall). Far below the 44px minimum.

**Fix:**
```tsx
<button className="px-2 py-1 -ml-2 rounded text-xs text-text-secondary hover:text-accent transition-colors">
  Balas
</button>
<button className="px-2 py-1 -ml-2 rounded text-xs text-text-secondary hover:text-bearish ...">
  Hapus
</button>
```

### U3. Ticker tag remove button is a literal "x" character — 8×12px touch target

**File:** `src/components/community/post-composer.tsx:167-174`

```tsx
<button type="button" onClick={() => setTickerTag(null)} className="hover:text-bearish transition-colors" aria-label="Hapus tag">
  x
</button>
```

The "x" character is approximately 8×12px. Even with the parent padding, the actual button element has no padding of its own. Users with motor impairments cannot reliably tap this.

**Fix:**
```tsx
<button type="button" onClick={() => setTickerTag(null)} className="p-1 -m-0.5 rounded hover:text-bearish hover:bg-bearish/10 transition-colors" aria-label="Hapus tag">
  <svg width="10" height="10" viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2.5" strokeLinecap="round"><line x1="18" y1="6" x2="6" y2="18"/><line x1="6" y1="6" x2="18" y2="18"/></svg>
</button>
```

### U4. Auth form inputs are only ~36px tall on mobile — below 44px minimum

**Files:** `src/app/auth/signin/page.tsx`, `register/page.tsx`, `complete-profile/page.tsx`

All inputs use `py-2` (8px padding). With 16px text that's ~32px total. iOS auto-zooms on inputs below 16px font-size when the viewport is scaled — but the touch target is still too small.

**Fix — change all auth input padding from `py-2` to `py-3`:**
```tsx
// All auth page inputs:
className="... py-3 ..."
```

This gives ~40px height, much closer to the 44pt minimum. Also change buttons from `py-2.5` to `py-3`.

### U5. Color-only meaning on bullish/bearish — no icon/text supplement

**Files:** `src/components/stock/stock-card.tsx:45`, `src/components/stock/key-statistics.tsx:80`

The stock card uses a colored dot (green/red) to indicate direction. The key statistics use "Above"/"Below" text with color. But the stock card dot is **color-only** — no text or icon to distinguish for colorblind users (~8% of men).

**Fix — add a micro arrow to the dot:**
```tsx
// stock-card.tsx, replace the dot span:
<span className={`flex items-center justify-center w-4 h-4 rounded-full text-white text-[8px] font-bold ${
  isPositive ? "bg-bullish" : "bg-bearish"
}`} aria-hidden="true">
  {isPositive ? "↑" : "↓"}
</span>
```

## High Impact — Touch & Interaction (Priority 2)

### U6. Stock table rows have no cursor-pointer on clickable ticker links

**File:** `src/components/stock/stock-table.tsx:185`

```tsx
<Link href={`/stocks/${stock.ticker}`} className="font-semibold text-accent hover:underline">
```

The entire row is clickable through this link, but only the ticker text shows underline on hover. The row background changes (`hover:bg-accent/[0.04]`) but there's no cursor change until you hover the ticker specifically. Users don't know the row is clickable.

**Fix — make the ticker link cover the full row:**
Use a `<Link>` wrapping the entire `<tr>` or add `cursor-pointer` to the row:
```tsx
<tr className={`transition-colors cursor-pointer ${isEven ? "bg-bg-card/40" : ""} hover:bg-accent/[0.04]`}
    onClick={() => window.location.href = `/stocks/${stock.ticker}`}>
```

### U8. Report button dropdown missing Escape key close

**File:** `src/components/community/report-button.tsx`

The report button closes on outside click but not on Escape key. Same pattern as the user dropdown (which we fixed in the previous audit).

**Fix:**
```tsx
useEffect(() => {
  if (!open) return;
  const handler = (e: KeyboardEvent) => { if (e.key === "Escape") setOpen(false); };
  document.addEventListener("keydown", handler);
  return () => document.removeEventListener("keydown", handler);
}, [open]);
```

### U9. Stock table sector pills overflow has no mobile indicator

**File:** `src/components/stock/stock-table.tsx:126-155`

On mobile, the sector pills scroll horizontally. There's no visual indicator that pills are hidden off-screen to the right. Users with 375px-wide screens see "Semua Perbankan Aneka" and might not realize there are 6+ more sectors.


### U12. Stock detail chart is 450px fixed height — too tall on small phones

**File:** `src/components/chart/candlestick-chart.tsx:242`

```tsx
<div ref={containerRef} className="w-full h-[450px]" ... />
```

On a 375×667 iPhone SE, 450px is 67% of the viewport. After the header card (~120px) and toolbar (~40px), the chart pushes all indicators and statistics below the fold with no indication they exist.

**Fix — use responsive height:**


## Medium — Typography & Color (Priority 6)

### U13. `text-[9px]` category badge on screener presets — below 10px minimum

**File:** `src/app/screener/page.tsx:34`

```tsx

```

9px text is illegible on non-retina displays and fails WCAG minimum text size. The "Momentum", "Trend", "Volume" labels are functional — they help users scan presets.

**Fix:**
```tsx
<span className="text-[10px] font-medium text-text-tertiary uppercase tracking-wider ...">
```

### U14. `text-[7px]` nowhere in codebase but `text-text-tertiary` at 10px fails contrast

The design token `--color-text-tertiary: #a8a29e` on white (`#ffffff`) has a contrast ratio of approximately 3.1:1. WCAG AA requires 4.5:1 for normal text. This is used extensively for labels, badges, and secondary info.

At `text-[10px]` and `text-[11px]`, this is normal text (not large), so 4.5:1 is required.

**Fix — darken text-tertiary:**
```css
/* globals.css */
--color-text-tertiary: #928b85;  /* was #a8a29e, now ~4.6:1 on white */
```

## Medium — Animation (Priority 7)

### U15. Post composer tag removal uses no animation

**File:** `src/components/community/post-composer.tsx:167`

When a ticker tag is removed, it disappears instantly. The surrounding content shifts to fill the gap with no transition. This violates `layout-shift-avoid` and `state-transition`.

**Fix — add transition:**
```tsx
<span className="inline-flex items-center gap-1 px-2 py-0.5 rounded-full bg-accent/10 text-accent text-xs font-medium transition-all duration-200">
```

And consider using `layoutId` or at minimum a CSS transition on the parent flex container.

## Medium — Forms & Feedback (Priority 8)

### U16. No autoFocus on first input in any auth form

**Files:** All auth pages

First input should receive `autoFocus` so users can start typing immediately without tapping.

**Fix:**
```tsx
<input autoFocus type="email" id="email" ... />
```

---

### U17. Auth form disabled state uses only opacity — no semantic feedback

**Files:** All auth pages

```tsx
<button disabled={isPending} className="... disabled:opacity-50">
```

Opacity alone isn't enough to signal "this button is disabled". Per Material Design, disabled state needs: reduced opacity (0.38–0.5) + cursor change + semantic `disabled` attribute.

**Fix:**
```tsx
<button disabled={isPending} className="... disabled:opacity-50 disabled:cursor-not-allowed">
```

Add `disabled:cursor-not-allowed` to all disabled buttons across auth forms.

## Low — Charts & Data (Priority 10)

### U19. Chart tooltip not keyboard-accessible

**File:** `src/components/chart/candlestick-chart.tsx`

The lightweight-charts library shows tooltips on hover/crosshair. These are not keyboard accessible. For WCAG compliance, the chart should have an `aria-label` with a text summary.

**Current:**
```tsx
<div ref={containerRef} className="w-full h-[450px]" role="img" aria-label="Stock price candlestick chart" />
```

**Fix — make the aria-label dynamic with key data:**
```tsx
<div
  ref={containerRef}
  className="w-full h-[320px] sm:h-[400px] lg:h-[450px]"
  role="img"
  aria-label={`Stock price chart for ${data.length > 0 ? `latest close ${data[data.length - 1].close}` : 'no data'}`}
/>
```

### U20. RSI coloring uses green/red only — colorblind users can't distinguish

**Files:** `src/components/stock/indicator-panel.tsx`, `src/components/stock/stock-table.tsx`

RSI overbought (red) vs oversold (green) is color-only meaning. ~8% of men have red-green colorblindness.

**Fix — supplement color with text labels (already done via SignalBadge "Overbought"/"Oversold") but ensure the table mini-bar also has a non-color indicator:**

For the stock table, the RSI value text is already colored AND shows the numeric value — this is acceptable. The mini-bar uses colored position indicators. Consider adding a thin center line at 50 to help orientation.
